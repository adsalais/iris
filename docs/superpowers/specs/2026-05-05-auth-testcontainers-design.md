# Auth integration tests with testcontainers — design

**Date:** 2026-05-05
**Scope:** Add a new integration tier under `tests/auth/integration/` that exercises the LDAP and OAuth providers (and their full route-level login flows) against real `bitnami/openldap:2.6` and `quay.io/keycloak/keycloak:26.0` containers managed by `testcontainers-python`. Covers happy paths and natural failure paths exercisable against a real IdP, with full TLS coverage for both providers.
**Out of scope:** Replacing the existing offline tests (`tests/auth/test_provider_ldap.py`, `tests/auth/test_provider_oauth.py`); they stay as-is. State-mismatch / missing-state-cookie OAuth-callback tests don't need a real IdP — they're called out here as a follow-up offline test gap, not part of the integration tier. Mutation-style tests (user disabled mid-session, group membership changed mid-session) are deferred; v1 integration tests are read-only against seeded directories/realms.

## Motivation

The current `tests/auth/test_provider_ldap.py` uses `ldap3.MOCK_SYNC` and `tests/auth/test_provider_oauth.py` uses `httpx.MockTransport`. Both are valuable: they're fast, offline, and exercise security edges that are mock-only by nature (DN injection, ldap3 typed-exception classification, forged-key id_tokens, locale-independent error classification). Those tests stay.

But three classes of confidence are unreachable from those mocks:

1. **Real-IdP interop.** Does our PKCE/state plumbing actually round-trip through Keycloak's authorize endpoint? Does our `groupOfNames` search syntax actually return the rows we expect from a real OpenLDAP server? The offline tests assume the spec; the integration tests verify the wire.
2. **TLS code paths.** `LDAP_REQUIRE_TLS=true` and `LDAP_CA_CERT_PATH` are essentially untestable today — `ldap3.MOCK_SYNC` doesn't speak TLS. Same for HTTPS Keycloak with self-signed certs. These are the configurations real deployments will use.
3. **End-to-end route flows.** The LDAP and OAuth login routes are not exercised end-to-end against any IdP today. Provider-level offline tests stop at `User` construction; nothing currently asserts that a real-IdP login produces the expected `session.roles` once the YAML role mapping has been applied.

The clickhouse test suite already establishes the testcontainers pattern in this repo (`tests/clickhouse/conftest.py`); this design extends it.

## High-level architecture

```
tests/auth/
├── test_provider_ldap.py             # existing offline tests (unchanged)
├── test_provider_oauth.py            # existing offline tests (unchanged)
├── test_login_mock.py / test_csrf.py / ...  (unchanged)
└── integration/
    ├── conftest.py                   # container fixtures, TLS certs, seed paths
    ├── seed/
    │   ├── ldap.ldif                 # alice/bob/carol, admins/users groups
    │   └── keycloak-realm.json       # iris-test realm, client, users, groups
    ├── _keycloak_helpers.py          # simulate_login(...) — drives Keycloak login form
    ├── _tls.py                       # generate_ca_and_leaf(...) — pure cryptography
    ├── test_ldap_integration.py      # provider + route tests against OpenLDAP
    └── test_oauth_integration.py     # provider + route tests against Keycloak
```

No `__init__.py` under `tests/auth/integration/` — `--import-mode=importlib` requires `tests/` to *not* be a package, in exchange for unique basenames across the suite. `test_ldap_integration.py` / `test_oauth_integration.py` are unique within `tests/`. Helper modules use a leading underscore so they aren't collected as test files. CLAUDE.md already documents the rule.

**Why a sub-package, not flat in `tests/auth/`:**
- Discoverability: `uv run pytest tests/auth/integration` runs only the integration tier; `uv run pytest --ignore=tests/auth/integration` excludes it on a Docker-less laptop.
- Containment: the integration `conftest.py` layers heavy session-scoped fixtures on top of `tests/conftest.py` without leaking them into the fast tests.

**Generated artifacts:** TLS certs are generated by a session-scoped fixture into `tmp_path_factory.mktemp("auth-certs")` and not committed. The LDIF and realm-import JSON files ARE committed — they're declarative seed data, easy for humans to review.

## Containers and seed data

### LDAP — `bitnami/openldap:2.6`

Driven via a generic `testcontainers.core.container.DockerContainer` wrapper (no first-party `testcontainers.openldap` module). The bitnami image accepts:

- `LDAP_ROOT=dc=corp,dc=local` — base DN for the seeded directory
- `LDAP_ADMIN_USERNAME=admin`, `LDAP_ADMIN_PASSWORD=adminpw` — bind user for any admin-API mutations (none in v1, but available)
- `LDAP_CUSTOM_LDIF_DIR=/ldifs` — directory of LDIFs applied at first boot. Our `ldap.ldif` is mounted there.
- `LDAP_TLS_CERT_FILE`, `LDAP_TLS_KEY_FILE`, `LDAP_TLS_CA_FILE` — paths to the generated leaf cert + key + CA (mounted from the host into the container).
- `LDAP_ENABLE_TLS=yes`, `LDAP_LDAPS_PORT_NUMBER=1636` — turn on `ldaps://`. Plain `ldap://` stays available on 389 for negative-path TLS tests.

Exposed ports: `1636` (ldaps) and optionally `389` (ldap) — bound to random host ports by testcontainers.

`tests/auth/integration/seed/ldap.ldif`:

`carol` exists but is in no group — used by the "user with no groups → groups=()" test. Mirrors the existing offline directory shape exactly so happy-path assertions stay parallel between offline and integration tiers.

### OAuth — `quay.io/keycloak/keycloak:26.0`

Driven via `testcontainers.keycloak.KeycloakContainer` if the extra is available in the pinned version of `testcontainers-python`; otherwise via a generic `DockerContainer` wrapper (~30 lines, see Risks). The container starts with `--import-realm` and our realm JSON mounted at `/opt/keycloak/data/import/iris-test-realm.json`. We also pass `--https-certificate-file` / `--https-certificate-key-file` so Keycloak serves HTTPS on its random host port. Plain HTTP stays available for the wrong-CA-bundle negative test.

`tests/auth/integration/seed/keycloak-realm.json` (abbreviated for the spec; the full file lives in `seed/`):

```json
{
  "realm": "iris-test",
  "enabled": true,
  "users": [
    {
      "username": "alice",
      "enabled": true,
      "credentials": [{"type": "password", "value": "secret"}],
      "groups": ["/admins", "/users"],
      "firstName": "Alice"
    },
    {
      "username": "bob",
      "enabled": true,
      "credentials": [{"type": "password", "value": "hunter2"}],
      "groups": ["/users"]
    }
  ],
  "groups": [{"name": "admins"}, {"name": "users"}],
  "clients": [
    {
      "clientId": "iris",
      "secret": "iris-test-secret",
      "redirectUris": ["http://testserver/login/callback"],
      "publicClient": false,
      "directAccessGrantsEnabled": false,
      "standardFlowEnabled": true,
      "protocolMappers": [
        {
          "name": "groups",
          "protocol": "openid-connect",
          "protocolMapper": "oidc-group-membership-mapper",
          "config": {
            "claim.name": "groups",
            "full.path": "false",
            "id.token.claim": "true",
            "userinfo.token.claim": "true"
          }
        }
      ]
    }
  ]
}
```

The explicit `oidc-group-membership-mapper` is **load-bearing**: Keycloak does not emit a `groups` claim in id_tokens or userinfo by default. Without this mapper the integration tests would observe `User.groups == ()` and we'd silently lose role-mapping coverage. Pinning the mapper in the seed also documents the configuration that real Keycloak deployments need.

`redirectUris: ["http://testserver/login/callback"]` matches `TestClient`'s default Host header (`testserver`). The Keycloak realm rejects redirect URIs that don't match exactly.

### TLS certs

A session-scoped fixture in `_tls.py` generates a CA + leaf cert in pure Python via `cryptography` (already a transitive dep through `pyjwt`/`authlib`):
- RSA-2048 keys, both CA and leaf
- CA: CN=`iris-test-ca`, validity 1 year
- Leaf: CN=`localhost`, SANs=`DNS:localhost,IP:127.0.0.1`, signed by the CA
- Output: three files in `tmp_path_factory.mktemp("auth-certs")` — `ca.pem`, `server.pem`, `server.key`. Permissions: `server.key` is `0600`.

The same leaf cert serves both OpenLDAP and Keycloak (they both bind to `localhost:<random-port>`). The CA bundle is mounted into both containers and consumed by the test's `ldap3` and `httpx` clients via:
- `LDAP_CA_CERT_PATH=str(ca_pem)` (read by `LDAPProvider.from_settings`)
- `_http_transport=httpx.AsyncHTTPTransport(verify=str(ca_pem))` injected into `OAuthProvider` via the existing test-injection seam.

No openssl shell-out, no committed certs.

## Per-test app construction

Each integration test gets its own `app` configured for LDAP or OAuth via `monkeypatch.setenv` overriding the `os.environ.setdefault("AUTH_METHOD", "mock")` baked in at `tests/conftest.py:10`. The existing `build_app()` factory at `src/iris/app.py:44` reads env at call-time, so each test gets a freshly-wired provider.

```python
@pytest.fixture
def ldap_app(monkeypatch, openldap_container, tls_paths):
    monkeypatch.setenv("AUTH_METHOD", "ldap")
    monkeypatch.setenv("LDAP_URL", openldap_container.ldaps_url)
    monkeypatch.setenv("LDAP_BIND_DN_TEMPLATE", "uid={username},ou=people,dc=corp,dc=local")
    monkeypatch.setenv("LDAP_GROUP_BASE_DN", "ou=groups,dc=corp,dc=local")
    monkeypatch.setenv("LDAP_REQUIRE_TLS", "true")
    monkeypatch.setenv("LDAP_CA_CERT_PATH", str(tls_paths.ca_pem))
    from iris.app import build_app
    return build_app()


@pytest.fixture
def oauth_app(monkeypatch, keycloak_container, tls_paths):
    monkeypatch.setenv("AUTH_METHOD", "oauth")
    monkeypatch.setenv("OIDC_ISSUER_URL", f"{keycloak_container.https_url}/realms/iris-test")
    monkeypatch.setenv("OIDC_CLIENT_ID", "iris")
    monkeypatch.setenv("OIDC_CLIENT_SECRET", "iris-test-secret")
    monkeypatch.setenv("OIDC_SCOPES", "openid profile email groups")
    from iris.app import build_app
    return build_app()
```

`monkeypatch` restores the prior env after the test, so subsequent unrelated tests in the session see `AUTH_METHOD=mock` again. No test pollution.

## Route-level OAuth orchestration

The canonical Authorization Code flow involves the user's browser visiting Keycloak's login form — TestClient can't follow a redirect to a different host. We solve this with one small helper in `_keycloak_helpers.py`:

```python
def simulate_login(*, test_client, http, username, password) -> httpx.Response:
    """Drive the OAuth Authorization Code flow against a real Keycloak.

    Returns the iris-side response that has just received the callback —
    i.e. the response you'd see after a successful login redirect chain.
    """
    # 1. iris -> Keycloak: 302 to authorize endpoint, sets oauth_state cookie
    r = test_client.get("/login", follow_redirects=False)
    authorize_url = r.headers["location"]

    # 2. user-agent visits Keycloak login page
    page = http.get(authorize_url)
    form_action = _extract_form_action(page.text)

    # 3. POST credentials; Keycloak responds with 302 to our callback
    submit = http.post(
        form_action,
        data={"username": username, "password": password},
        follow_redirects=False,
    )
    callback_url = submit.headers["location"]

    # 4. browser hits our callback with code+state, carrying oauth_state cookie
    return test_client.get(callback_url, follow_redirects=False)
```

`http` is an `httpx.Client(verify=str(tls_paths.ca_pem))` for TLS to the self-signed Keycloak. `_extract_form_action` is a deliberately small regex (`r'<form[^>]*\baction="([^"]+)"'`) — coupling to Keycloak's login HTML is concentrated in this one place. A future Keycloak major bump that changes the form layout is a one-line fix.

Provider-level OAuth tests reuse the helper to obtain a fresh authorization code, then stop after step 3 and feed that code to `OAuthProvider.exchange_code(...)` directly.

**Risk** (accepted): Keycloak login HTML is stable across minor versions but can shift across major versions. Mitigation: pin the Keycloak image tag the same way clickhouse pins `26.3`, and keep the parser small.

## Test plan

Total: ~17 new tests across two files. The integration tier covers what *requires* a real IdP. Anything testable offline (DN injection, forged-key id_tokens, locale-specific exception classification, oauth_state cookie Secure-flag handling, etc.) stays in the existing offline tests.

### `test_ldap_integration.py`

**Provider layer (~6 tests)**
- `test_authenticate_alice_returns_groups_from_real_directory` — `LDAPProvider.authenticate("alice", "secret")` → `User(subject="uid=alice,...", username="alice", display_name="Alice")`, `set(user.groups) == {"admins", "users"}`
- `test_authenticate_bob_returns_only_users_group` — `set(user.groups) == {"users"}`
- `test_authenticate_carol_with_no_group_membership` — carol exists, is in no group → `user.groups == ()`
- `test_authenticate_with_wrong_password_raises_invalid_credentials`
- `test_authenticate_with_unknown_user_raises_invalid_credentials`
- `test_ldaps_with_wrong_ca_bundle_raises_unreachable` — point `LDAP_CA_CERT_PATH` at a *second* generated CA → bind raises certificate-verify-error → surfaces as `AuthError("ldap_unreachable")`

**Route layer (~3 tests)**
- `test_login_alice_creates_session_with_admin_role` — `POST /login alice/secret` → 302 + `iris_session` cookie; `GET /api/whoami` → `{reader, writer, admin} ⊆ session.roles` (per the test authz YAML's `admins: [admins]` → `admin` mapping with `includes: [writer]` → `includes: [reader]`)
- `test_login_bad_password_redirects_with_error_token` — 302 to `/login?error=invalid_credentials&next=/`, no session cookie set
- `test_login_unknown_user_redirects_with_error_token` — same redirect

### `test_oauth_integration.py`

**Provider layer (~6 tests)**
- `test_exchange_code_returns_alice_with_groups` — happy path: drive `simulate_login`, intercept the code, call `provider.exchange_code(...)` → `User(username="alice", groups={"admins","users"})`
- `test_exchange_code_returns_bob_with_users_group_only`
- `test_exchange_code_with_wrong_client_secret_raises_oauth_exchange` — separate `OAuthProvider` instance configured with `OIDC_CLIENT_SECRET="wrong"`
- `test_exchange_code_with_mismatched_redirect_uri_raises_oauth_exchange` — code obtained against `http://testserver/login/callback`, exchanged with `http://testserver/different`
- `test_exchange_code_reuse_raises_oauth_exchange_on_second_call` — first call succeeds, second call against the same code fails (Keycloak invalidates on first use)
- `test_discovery_with_wrong_ca_bundle_raises_oauth_discovery` — `OAuthProvider` constructed with an httpx transport that verifies against a *different* CA bundle → first property access (which triggers discovery) raises `AuthError("oauth_discovery")`

**Route layer (~2 tests)**
- `test_oauth_login_alice_via_full_flow_creates_session_with_admin_role` — `simulate_login(...)` returns the iris-side response containing `iris_session` cookie; `GET /api/whoami` → admin role
- `test_oauth_login_bob_creates_session_with_no_roles` — bob is only in `users`, which doesn't map to a role in the test authz YAML, so `session.roles == frozenset()`

### Out of integration scope (offline gap, called out separately)

State-mismatch and missing-state-cookie cases for `GET /login/callback` don't need a real IdP. They should be added as offline route-level tests (`tests/auth/test_login_oauth.py`?) but are not part of this work item.

## Per-test isolation

All v1 integration tests are read-only against the seeded directory/realm. No UUID-prefixed entities (unlike clickhouse, where DDL mutation is the whole point). Containers are session-scoped; mirroring the clickhouse pattern keeps the boot cost paid once per pytest run.

The OAuth code-reuse test consumes a code that Keycloak invalidates internally; subsequent tests fetch fresh codes via `simulate_login`. No test-level cleanup required.

If a future test needs mutation (user disabled mid-session, group membership changed mid-test), it would seed via `ldap3.add` / Keycloak admin API against UUID-prefixed entities. Out of scope for v1 but not foreclosed.

## Docker behavior

Match the clickhouse precedent: testcontainers raises if Docker is unavailable; no auto-skip. To exclude the integration tier on a Docker-less laptop:

```
uv run pytest --ignore=tests/auth/integration
```

This keeps the failure mode loud and consistent across `tests/clickhouse/` and `tests/auth/integration/`. A softer `pytest_collection_modifyitems` auto-skip-when-Docker-unreachable hook is rejected for v1 — silent skips can mask CI misconfiguration.

## Dependencies

- **`testcontainers[keycloak]`** (dev) — verifies the `keycloak` extra is published on the version of `testcontainers-python` already in the lockfile. If not, fall back to a generic `DockerContainer` wrapper (~30 lines) that exposes `get_container_host_ip()` + `get_exposed_port(8443)` and constructs the `https_url`.
- **`cryptography`** — already transitive via `pyjwt` / `authlib`. No new top-level dep.
- **`httpx`** — already in tree.

The `bitnami/openldap` and `quay.io/keycloak/keycloak` images are *not* repository dependencies; they're pulled by Docker on first run, the same way `clickhouse/clickhouse-server:26.3` is.

## Runtime cost (estimated)

- Keycloak boot: ~25s (cold pull adds ~30s the first time on a new machine)
- OpenLDAP boot: ~5–10s
- Test execution: ~17 tests at <100ms each ≈ 2s
- Total integration tier: **~45–60s** per `pytest` invocation. Session-scoped containers amortize across the full suite.

The offline tier stays unchanged at ~5s. A developer running `uv run pytest --ignore=tests/auth/integration` keeps the fast tier; CI runs everything.

## Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Keycloak login form HTML changes across major versions | Medium (on KC bump) | One-line fix in `_extract_form_action` | Pin image tag to `26.0`; concentrate parsing in one helper |
| `testcontainers[keycloak]` extra not published in the pinned testcontainers-python version | Low–medium | Fall back to generic `DockerContainer` (~30 lines) | Verify in implementation phase 0; design accepts either path |
| Realm-import JSON drifts from what Keycloak 26.x accepts | Low | Container fails to boot with clear log | Pin major version; use realm-export from a real KC as a starting point |
| Docker-on-CI flakiness (port races, etc.) | Low–medium | Flaky run | Standard testcontainers behavior; same exposure as clickhouse tier |
| Integration tier slows the developer feedback loop | High (always) | ~45–60s added to a pytest run | Sub-package + `--ignore=...` gives an opt-out; offline tier stays fast |

## Migration notes / order of work

The implementation plan (handed off to writing-plans) will sequence these:

1. Add the dev dep `testcontainers[keycloak]`; verify Phase-0 that an empty container fixture boots both images.
2. Generate TLS infrastructure in `_tls.py`; verify `httpx` and `ldap3` accept the generated CA + leaf.
3. Author seed files (`ldap.ldif`, `keycloak-realm.json`); boot containers with seeds; verify a one-shot bind / discovery works from the test process.
4. Write `_keycloak_helpers.simulate_login`; smoke-test it end-to-end before any real assertion.
5. Provider-layer LDAP tests (~6).
6. Provider-layer OAuth tests (~6).
7. Route-layer LDAP tests (~3).
8. Route-layer OAuth tests (~2).
9. Document the `--ignore=tests/auth/integration` opt-out and the runtime cost in CLAUDE.md.

Each phase is small enough to land independently, with the test suite green throughout.
